# 行程规划用户分类与引导流程 PRD

## 目标

在 `行程规划` 模块中，先按用户的规划成熟度分流，而不是先按年龄、身份或旅行标签分流。这样可以让页面入口、AI 提问和结果生成方式都更贴合用户当前状态。

## 为什么这样分

- 同一个用户可能是家庭用户、情侣用户、亲子用户，但真正决定产品流程的是他已经规划到了哪一步。
- 规划成熟度决定系统应该先做什么。
- 有些用户需要导入和整理，有些用户需要共同生成，有些用户需要先被推荐目的地。

## 用户分类

### 1. 已有详细计划

特征：
- 已经整理了城市、日期、酒店或景点顺序
- 可能已经有文档、表格、聊天记录或截图
- 只差补时间轴、查冲突、转路书

入口文案：
- 我已经有比较完整的计划，想把它整理成可执行路书

按钮文案：
- 导入我的计划

首轮问题：
- 你的行程现在是什么形式：文字、表格、聊天记录还是图片？
- 这次旅行一共几天，出发日期是什么？
- 你最希望系统补齐什么：时间安排、路线优化还是执行路书？

补充字段：
- 已有行程内容
- 已订酒店/交通
- 不可改动项
- 希望系统补齐的内容

输出结果：
- 结构化行程
- 时间轴补齐
- 冲突检查
- 动态路书

### 2. 已有想法，未成行程

特征：
- 已经知道想去哪几个地方
- 已经有天数、偏好、约束
- 还没有完整路线和每天安排

入口文案：
- 我已经有大概方向，想一起把行程搭出来

按钮文案：
- 一起生成草案

首轮问题：
- 你已经确定想去哪些地方？
- 一共计划玩几天？
- 这次更偏轻松、均衡还是高密度？
- 有没有特殊约束，比如老人、小孩、不自驾、少换酒店？

补充字段：
- 已确定目的地列表
- 必去点
- 可舍弃点
- 是否接受换酒店

输出结果：
- 行程骨架
- 住宿锚点
- 城市顺序
- 逐日 V1 草案
- 局部修改入口

### 3. 只确定想去的地方

特征：
- 只锁定了一个景点或目的地
- 不知道玩几天、住哪里、是否联动周边

入口文案：
- 我想去某个地方，但还不知道怎么安排最合适

按钮文案：
- 围绕目的地规划

首轮问题：
- 你最想去的地方是哪里？
- 这次大概有几天时间？
- 你想深度玩一个地方，还是顺带串联周边？
- 你更在意风景、人文、美食还是轻松度？

补充字段：
- 核心目的地
- 是否接受周边联动
- 希望玩法类型
- 可接受移动强度

输出结果：
- 2 到 3 种玩法方向
- 推荐时长
- 周边联动建议
- 选中方向后进入行程骨架生成

### 4. 只有旅行需求，未定目的地

特征：
- 只有预算、天数和旅行愿望
- 还未决定去哪里

入口文案：
- 我还没决定去哪里，想先找到合适的旅行方向

按钮文案：
- 帮我推荐方向

首轮问题：
- 你从哪里出发，什么时候出行？
- 预算大概多少，计划几天？
- 同行人是谁：情侣、家庭、亲子还是带长者？
- 你更想要休息放松、风景体验、城市漫游还是深度旅行？

补充字段：
- 旅行动机
- 气候偏好
- 是否出境
- 风格偏好

输出结果：
- 目的地方向推荐
- 方向对比
- 用户确认后进入后续规划

## 通用基础字段

所有类型共用的基础字段：
- 出发地
- 出行日期
- 天数
- 同行人类型
- 预算区间
- 交通偏好
- 旅行节奏
- 特殊约束

## 酒店在规划中的角色

酒店不是单纯的执行资源，它也是规划变量。

在行程规划阶段，系统应该先感知：
- 某个住宿锚点的大致价格带
- 当前日期的库存紧张程度
- 哪个片区更适合作为住宿落点
- 是否存在更合理的替代片区

也就是说：
- 行程规划阶段：做住宿价格感知和风险提示
- 动态路书阶段：做正式预订执行和订单回填

## 住宿成本感知

建议在 `行程规划` 页面增加一个 `住宿成本感知` 模块。

第一阶段先不追求真实下单能力，而是提供：
- 住宿锚点建议
- 区域级价格带
- 风险等级
- 替代片区建议
- 对路线的影响说明

示例：
- 丽江古城：￥500 - 800 / 晚，风险中
- 束河：￥350 - 600 / 晚，风险低
- 白沙：￥600 - 1000 / 晚，风险高

产品价值：
- 如果某片区太贵，用户可能直接不去
- 如果某片区库存紧张，系统应提示尽早锁定
- 如果某替代片区更便宜且不影响次日出发，应优先推荐

## 页面建议

首页先展示 4 张入口卡片。每张卡片包含：
- 类型标题
- 适用描述
- 入口按钮
- 3 到 4 个首轮问题预览
- 该类型的补充字段标签

用户选定一种类型后，再进入对应的详细收集流程。

同时在行程规划页中加入：
- 住宿成本感知模块
- 酒店价格对路线影响的说明
- 替代片区示例

## AI 角色建议

- 已有详细计划：整理员 + 校对员
- 已有想法，未成行程：共创规划师
- 只确定想去的地方：路线设计师
- 只有旅行需求，未定目的地：旅行顾问

## 结果形态建议

不要让所有用户都拿到同一种结果。

- 已有详细计划：先拿到时间轴和冲突修正
- 已有想法，未成行程：先拿到骨架和逐日草案
- 只确定想去的地方：先拿到玩法方向
- 只有旅行需求，未定目的地：先拿到目的地推荐

## 上线优先级

1. 已有想法，未成行程
2. 已有详细计划
3. 只确定想去的地方
4. 只有旅行需求，未定目的地

## 原因

- 前两类离路书执行最近
- 最容易接现有动态路书系统
- 最容易形成规划到执行的闭环
- 后两类更依赖推荐和探索能力，可以后续逐步补强
